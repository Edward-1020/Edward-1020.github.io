你必须懂的前端安全（二）- CSRF

发表于 2018-03-12 | 分类于 web前端安全

CSRF

  • Cross Site Request Forgy
  • 跨站请求伪造
  • 与XSS区别是：XSS是指本网站运行了来自其它网站的脚本，及非本网站的脚本。而CSRF是指我在其它的网站对本网站产生了影响。即在其它的网站对目标网站发出了一些请求，这些请求是在用户不知情的情况下完成的。
  • 危害：我打开了一些网站，却发现我的钱没了。

举例

POST

  • 提交表单的时候页面会跳转，但该页面不会跳转。因为target=”csrf” 和 iframe的name也是“csrf”时，表单会在iframe中跳转。
  • iframe被隐藏了，用户是看不到自己被恶意脚本代表了，被动的发了一个请求。

GET

点击

如果网站允许get提交表单，那么CSRF的攻击会更加简单。

用户点击，就会发送get请求。

不点击

用户不用点击，也可以发送get请求。

CSRF原理

  1. 用户在目标网站a.com进行登录。确认身份，目标网站的后台将cookie种给浏览器。
  2. 恶意网站b.com通过手段向a.com发送数据。
  3. 虽然是从b.com发送的请求，但是带有a网站的cookie。
  4. 这样，用户在不知情的情况下，被b用自己的身份向a发送了一个请求。

危害

  • 利用用户的登录态：盗取用户资金。
  • 用户不知情：冒充用户发帖，损坏公司声誉。
  • 完成业务请求。

案例

分享
用户点击一个病毒网站，身份被代理，再发一条病毒分享。这样传播越来越快。

防御

防御的重点在攻击的原理。恶意网站发送请求的特征是：
1.B网站向A网站发送请求。
2.带A网站的cookies。
3.不访问A网站前端。
4.referer为B网站，表示请求的来源。

针对带A网站的cookies特性，禁止第三方网站带cookie

same-site属性，只有来自同一站点的请求，才能带有same-site的cookie。
sameSite=Strict:一个网站通过链接到了我们的网站，这个请求是不允许带cookie。
sameSite=Lax:链接可以带，但是ajax不可以带cookie。
但是这个sameSite属性的兼容性不好。

针对不访问A网站前端特性，在前端页面加入验证信息

图形验证码，因为验证码是随机的，所以在恶意网站无法知道目标网站提供给用户的验证码是什么。

token

  • 验证码对用户体验是不友好的，你不可能所有的请求都要有验证码，所以使用token。token的获取必须要经过目标网站的前端，由前端拿到随着请求提交，而攻击者不经过前端就没办法获得token，就会攻击失败。
  • 做法：提交表单必须经过我们的前端
  1. 用户访问某个表单页面。
  2. 此时针对这个表单页面，服务端生成一个Token，放在用户的Session中，或者浏览器的Cookie中。
  3. 前端在页面表单附带上Token参数。
  4. 用户提交请求后， 服务端验证表单中的Token是否与用户Session（或Cookies）中的Token一致，一致为合法请求，不是则非法请求。
  • 缺点： 当打开多个表单页面，前面的表单页面在cookie中用的token会失效，别后面的页面的表单页面所用的token覆盖了。

验证referer

通过referer验证请求是来自于自己的网站还是恶意网站。
